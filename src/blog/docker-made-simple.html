Docker made simple: what containers actually are

by Ayush  ·  4 min read

Everyone who explains Docker reaches for the shipping container metaphor. I'll skip it. Here's the actual problem Docker solves, and why it matters.

The problem: "it works on my machine"

You write some code. It runs perfectly on your laptop. You send it to a teammate. It crashes. You spend an hour debugging and discover their machine has a different version of Python, or a different library version, or a different operating system configuration.

This is the classic "it works on my machine" problem, and it's been annoying developers for decades. Docker is the most widely adopted solution to it.

What a container actually is

A container is a package that includes everything your code needs to run — the code itself, the runtime (like Python or Node.js), all the libraries it depends on, and the configuration. When you run a container, you're running that entire self-contained environment, not just your code on top of whatever happens to be installed on the host machine.

The result: the container runs exactly the same way on your laptop, on your teammate's machine, on the staging server, and in production. Same code, same dependencies, same behavior.

Images vs containers

This trips people up. They're related but different.

  • An image is a blueprint — a read-only snapshot of the environment. Think of it like a recipe.
  • A container is a running instance of an image. Think of it like a dish made from that recipe.

One image can create many containers. You can run the same image five times and get five isolated, identical containers.

The Dockerfile

A Dockerfile is a text file that describes how to build a Docker image. Here's a real one for a simple Python app: 

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

CMD ["python", "app.py"]

Reading top to bottom: start from the official Python 3.12 image, set the working directory to /app, copy and install dependencies, copy the rest of the code, and define the command to run when a container starts. That's it.

Three commands that cover most of what you'll do day to day

# Build an image from a Dockerfile in the current directory
docker build -t my-app .

# Run a container from that image
docker run my-app

# List running containers
docker ps

Docker vs a virtual machine

A VM creates a whole separate operating system on top of your machine. It's heavy and slow to start. A container shares the host machine's OS kernel — it only packages what's different from a standard Linux environment. This makes containers lightweight and fast to start (seconds, not minutes).

That's the foundation. Once you get comfortable with docker build and docker run, Kubernetes starts to make a lot more sense — because all it does is manage containers at scale. See the Kubernetes post if you want to go there next.
