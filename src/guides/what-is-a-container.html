What is a container, actually?

by Ayush  ·  6 min read

You'll hear "containerize your app" constantly in DevOps conversations. This guide explains what that actually means, what problem it solves, and what you need to know to start using Docker today.

Start with the problem

Imagine you write a Python script on your laptop. It works perfectly. You send it to a colleague. They run it and get an error. After 20 minutes of debugging, you discover they have Python 3.9 and you have Python 3.12, and one of your libraries behaves differently between the two versions.

This happens constantly. "It works on my machine" is one of the oldest complaints in software. Containers are the solution the industry settled on.

What a container packages

A container bundles your code together with everything it needs to run:

  • The runtime (Python 3.12, Node.js 20, whatever)
  • All your libraries and dependencies at their exact versions
  • Any system-level configuration your code depends on

That whole bundle runs as an isolated unit. When someone else runs your container, they're running your exact environment — not theirs. No version mismatch. No "works on my machine."

Images and containers

These two words trip people up. They're related but distinct.

An image is the blueprint — a read-only snapshot of the environment. A container is what you get when you run an image. The image doesn't change; you can run it a hundred times and get a hundred separate, identical containers.

Think of an image like a recipe and a container like the dish you made from it. Same recipe, same dish every time.

Writing a Dockerfile

A Dockerfile is a text file that describes how to build an image. Here's a minimal one: 

FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

CMD ["python", "app.py"]

Line by line:

  • FROM python:3.12-slim — start from an official Python 3.12 base image (Docker Hub has thousands of these for every language and framework)
  • WORKDIR /app — set the working directory inside the container
  • COPY requirements.txt . && RUN pip install — install dependencies before copying the rest of your code (Docker caches this layer, so rebuilds are faster)
  • COPY . . — copy the rest of your code in
  • CMD — the command to run when a container starts

The three commands you'll use every day

# Build an image from your Dockerfile
docker build -t my-app:latest .

# Run a container from that image
docker run my-app:latest

# See what containers are currently running
docker ps

Containers vs virtual machines

Both solve the "different environments" problem, but differently. A virtual machine runs a full operating system — it's heavy (gigabytes), slow to start (minutes), and uses a lot of resources. A container shares the host OS kernel and only packages what's different from the base. It's lightweight (megabytes), starts in seconds, and you can run dozens on the same machine.

Where to go next

Once you're comfortable with docker build and docker run, the natural next step is Docker Compose — for running multiple containers together (like your app + a database). After that, Kubernetes — for managing containers across multiple machines in production.

But start here. Run a container. Break it. Fix it. That's how it sticks.
